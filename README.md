## <img src="https://media.giphy.com/media/iY8CRBdQXODJSCERIr/giphy.gif" width="28"> About Me

I'm a backend engineer based in Indonesia. Most of what I build lives on the server side. The APIs, the data models, and the auth layers that keep business software actually running in production. I care less about which framework is trendy and more about whether a system stays correct when traffic climbs and parts of it start to fail.

A lot of my work has been around real business problems. Centralized authentication, inventory and warehouse systems, ERP-style platforms, and more recently AI features that have to be grounded in real company data instead of just sounding smart.

Here's the kind of work I enjoy most:

- 🧩 **Scaling things that get busy.** Designing services that stay predictable as data, users, and tenants grow.
- 🏛️ **Modeling messy domains.** Turning complicated business rules into something clean enough to maintain.
- ⚙️ **Killing manual work.** Replacing spreadsheets and repetitive operations with software people can trust.
- 🚀 **Making teams faster.** Building the tooling and abstractions that take friction out of everyone's day.

> I try not to just ship features. I try to ship systems that are still understandable a year later.

### 🎯 Current Focus

```text
Distributed systems    : service boundaries, messaging, and staying consistent
AI agents              : tool-using agents and multi-step reasoning pipelines
RAG architectures      : retrieval-augmented generation over real company data
Auth platforms         : centralized identity, RBAC, and SSO across services
Clean architecture     : domain-driven, dependency-inverted backend design
Multi-tenant systems   : tenant isolation and data partitioning at scale
```

## <img src="https://media.giphy.com/media/dWesBcTLavkZuG35MI/giphy.gif" width="28"> Featured Projects

<details>
<summary><b>🛰️ ATLAS · Centralized Authentication & Authorization Platform</b></summary>

<br/>

**Project Overview**

ATLAS is a centralized identity platform that handles authentication, authorization, and access governance for a whole set of internal services. The idea is simple. Product teams shouldn't each reinvent login and permissions. ATLAS owns that, and exposes a clean API for login, token issuance, role resolution, and fine-grained permission checks.

| Category | Details |
| :--- | :--- |
| **Purpose** | Give a fleet of internal services one source of truth for auth. |
| **Tech Stack** | FastAPI, PostgreSQL, Redis, Docker, JWT, OAuth2 |
| **Architecture Style** | Clean architecture, service-oriented, stateless auth core |
| **Security** | JWT with rotation, RBAC and ABAC, hashed credentials, session revocation, scoped tokens |
| **Scalability** | Stateless token validation, Redis-backed caching, horizontally scalable API nodes |
| **Business Impact** | Removes duplicated auth code and makes spinning up a new service much faster |

**Engineering Highlights**
- Stateless verification so any service can validate a token without calling back to a central store.
- Redis-backed permission and session cache to keep authorization checks fast under load.
- Policy-driven RBAC and ABAC that keeps access rules out of application code.

**Potential Production Use Cases**
- Single sign-on across an internal microservice ecosystem.
- Access governance for a platform with many tenants and roles.

</details>

<details>
<summary><b>🤖 ATABOT 2.0 · AI Assistant Grounded in Company Knowledge</b></summary>

<br/>

**Project Overview**

ATABOT 2.0 is a production RAG platform that turns scattered company data into something you can actually ask questions to. It ingests structured and unstructured sources, embeds them into a vector store, and grounds every LLM answer in retrieved context so responses come with sources instead of guesses.

| Category | Details |
| :--- | :--- |
| **Purpose** | Give teams grounded, context-aware AI answers over their own data. |
| **Tech Stack** | FastAPI, PostgreSQL, Vector Database, OpenAI, Redis, Docker |
| **Architecture Style** | Retrieval-augmented generation with modular ingestion and retrieval services |
| **Security** | Scoped data access, tenant-aware retrieval, API key management |
| **Scalability** | Async ingestion, cached embeddings, pluggable model providers |
| **Business Impact** | Cuts the time people spend hunting for answers buried in internal docs |

**Engineering Highlights**
- Modular ingestion pipeline that normalizes very different sources into clean, embeddable chunks.
- Grounding layer that anchors answers to retrieved sources to keep hallucination low.
- Provider-agnostic LLM abstraction so swapping models doesn't mean rewriting business logic.

**Potential Production Use Cases**
- Internal knowledge assistant for support, ops, and onboarding.
- Document-grounded Q&A over policies, manuals, and databases.

</details>

<details>
<summary><b>🎫 Ticket Queue System · High-Throughput Distributed Queue</b></summary>

<br/>

**Project Overview**

A distributed queueing system built to soak up traffic spikes and process requests fairly without dropping work. It separates taking requests in from actually processing them using a message-driven design, so the front door stays responsive even when the workers behind it are slammed.

| Category | Details |
| :--- | :--- |
| **Purpose** | Handle big bursts of requests without losing any of them. |
| **Tech Stack** | Go or FastAPI, NATS JetStream, Redis, PostgreSQL, Docker, K6 |
| **Architecture Style** | Event-driven, producer and consumer, backpressure-aware |
| **Security** | Authenticated producers, validated payloads, rate limiting |
| **Scalability** | Horizontal worker scaling, durable streams, load-tested with K6 |
| **Business Impact** | Keeps processing fair and the system standing when demand surges |

**Engineering Highlights**
- NATS JetStream durable streams so intake and processing are decoupled and survive worker restarts.
- Backpressure and fair ordering so no request gets starved under burst load.
- A K6 load-testing harness that checks throughput and latency before anything ships.

**Potential Production Use Cases**
- Event ticketing, virtual waiting rooms, or flash-sale queues.
- Any workload that needs a reliable buffer between fast intake and slower processing.

</details>
